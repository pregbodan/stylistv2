HARDDX PROJECT COMPLETE REPORT
==============================

Project name:
HARDDX - Intelligent AI Chatbot for Computer Hardware Diagnosis and Troubleshooting

Purpose:
This project is a hybrid hardware-diagnosis chatbot that takes a user's text
description of a computer problem, optionally combines it with an image-based
analysis result from Gemini Vision, then returns a fault category, likely
causes, troubleshooting steps, and a guided multi-turn repair flow.

The system is built as a Flask web app with a browser-based chat UI, a
rule-based knowledge base, a supervised text classifier, and a persistence
layer for dialogue state and conversation logging.


1. High-Level Project Summary
-----------------------------

HARDDX is designed to help users diagnose common computer hardware issues such
as power supply failures, overheating, RAM faults, storage failures, display
or GPU problems, and peripheral issues.

The application supports:
- Text-only diagnosis.
- Image-assisted diagnosis using Gemini Vision from the browser.
- Multi-turn clarifying questions before a diagnosis is finalized.
- Step-by-step troubleshooting guidance after diagnosis.
- Session persistence for dialogue state.
- Conversation logging in SQLite or MySQL.

The project structure shows it is intended as both an academic/demo system and
a practical diagnostic assistant.


2. Core Architecture
--------------------

The main pipeline is:

User message
 -> NLP preprocessing
 -> ML classifier + rule-based keyword matching
 -> confidence fusion
 -> diagnosis
 -> guided troubleshooting

Main components:
- app.py
  Flask application entry point and API routes.
- dialogue_manager.py
  Multi-turn state machine that handles clarifying, diagnosis, and guided
  troubleshooting stages.
- inference_engine.py
  Hybrid diagnostic engine that combines the supervised model and knowledge
  base keyword matching.
- train_model.py
  Trains the supervised text classifier.
- database.py
  Stores conversations and dialogue state.
- utils/nlp_preprocessor.py
  NLP preprocessing and entity extraction.
- templates/index.html
  Chat UI.
- static/js/chat.js
  Front-end chat behavior.
- static/js/vision.js
  Client-side Gemini Vision image analysis.
- static/css/style.css
  Console-style UI styling.


3. Application Flow
-------------------

3.1 Startup
- Flask app starts in app.py.
- database.init_db() ensures the database tables exist.
- DialogueManager is instantiated, which in turn loads:
  - the trained ML model from models/intent_classifier.pkl
  - the label list from models/label_encoder.pkl
  - the model name from models/model_name.pkl
  - the knowledge base from data/knowledge_base.json

3.2 User interaction
- The user types a problem into the chat UI.
- The front end can optionally attach an image.
- If an image is attached, static/js/vision.js may analyze it in the browser
  with Gemini Vision and produce a description plus suspected category.
- The browser sends the message and optional image_finding to /api/chat.
- The server passes the request to DialogueManager.
- DialogueManager decides whether to:
  - greet conversationally,
  - ask a clarifying question,
  - present a diagnosis,
  - move to the next guided step,
  - resolve the conversation, or
  - escalate to professional repair advice.

3.3 State handling
- Session state is stored per browser session in the dialogue_sessions table.
- This allows the app to remember whether it is in idle, clarifying,
  diagnosed, or guiding mode.


4. Front-End Chat UI
--------------------

The UI is a hardware-console style interface.

Visible pieces:
- Top bar with HARDDX branding.
- Vision key settings button.
- Sidebar with diagnostic methods and suggested prompts.
- Main console chat feed.
- Composer area for typing and file upload.
- Image preview bar for uploaded images.
- Modal for entering a Gemini API key.

Front-end behavior in static/js/chat.js:
- Appends user and assistant messages to the feed.
- Shows typing indicators while the backend or image analysis is working.
- Handles image preview and removal.
- Sends chat history along with the current message.
- Stores visible session history in localStorage.
- Renders a session-history panel in the sidebar.
- Allows the settings modal to open, save, close on overlay click, and close
  with Escape.


5. Image Diagnosis
------------------

Image diagnosis is client-side only.

How it works:
- The user clicks the attach button and selects an image.
- static/js/vision.js reads the file in the browser.
- The image is sent directly to Gemini Vision from the browser using the
  user's own Gemini API key.
- The response is parsed into:
  - description
  - suspected_category
- That result is merged with the text diagnosis path on the server.

Important detail:
- The server does not call Gemini directly.
- The user's API key is stored only in browser localStorage.

Supported suspected image categories in vision.js:
- power_supply_failure
- overheating
- ram_failure
- storage_failure
- display_gpu_failure
- peripheral_issue


6. Dialogue Manager
-------------------

dialogue_manager.py implements the multi-turn conversation logic.

Key states:
- idle
- clarifying
- diagnosed
- guiding

Behavior:
- On a fresh issue, it starts diagnosis.
- If the knowledge base defines clarifying questions, it asks up to two.
- Answers are added back into the text and re-run through diagnosis.
- After a diagnosis, it presents the first troubleshooting step.
- During guided steps:
  - "yes" or equivalent resolves the conversation.
  - "no" or equivalent moves to the next step.
  - Once steps are exhausted, the user is told to see a technician.

Small-talk protection:
- Messages like hi, hello, hey, thanks, and thank you are treated as
  conversational and should not trigger diagnosis.

History context:
- The dialogue manager can receive conversation history.
- It folds prior turns into the diagnostic context when appropriate.


7. Hybrid Inference Engine
--------------------------

inference_engine.py is the central reasoning layer.

It combines:
1. Supervised ML classification.
2. Rule-based keyword matching.
3. Confidence fusion.

7.1 Rule-based matching
- Each knowledge-base category has keywords.
- The engine counts keyword hits in the user message.
- The best keyword match becomes the rule category.

7.2 ML prediction
- Text is preprocessed with utils/nlp_preprocessor.py.
- A TF-IDF representation is created.
- The classifier predicts a fault category.
- Confidence is estimated from either:
  - predict_proba for Naive Bayes, or
  - softmax-normalized decision function scores for Linear SVM.
- A vocabulary-overlap guard prevents trusting totally unrelated text.

7.3 Fusion logic
- If ML and rule agree, confidence is boosted.
- If rule evidence is strong, it overrides a weaker ML guess.
- If only ML has signal, it can still return a prediction.
- If no signal is strong enough, the engine returns no_match.

7.4 Conversational categories
- Greeting and thanks are treated as conversational categories.
- This prevents greetings from being diagnosed as hardware faults.


8. Model Training
-----------------

The supervised model is trained in train_model.py.

Training data:
- data/hardware_faults_dataset.csv

Training label set:
- display_gpu_failure
- greeting
- overheating
- peripheral_issue
- power_supply_failure
- ram_failure
- storage_failure
- thanks

Training pipeline:
- Text preprocessing with utils/nlp_preprocessor.preprocess
- TF-IDF vectorization with n-grams 1 to 2
- Two candidate classifiers:
  - Multinomial Naive Bayes
  - Linear SVM (LinearSVC)

Evaluation:
- 80/20 train-test split
- 4-fold cross-validation on the training split
- Accuracy and classification report printed for both candidates

Selection rule:
- The model with the best mean CV accuracy is selected.
- That model is then refit on the full dataset.
- The trained pipeline is saved to models/intent_classifier.pkl
- The sorted label list is saved to models/label_encoder.pkl
- The chosen model name is saved to models/model_name.pkl


9. Trained Model Results
------------------------

Based on the contents of train.txt, the training run produced:

Naive Bayes:
- CV accuracy: 0.588 +/- 0.041
- Held-out test accuracy: 0.700

Linear SVM:
- CV accuracy: 0.650 +/- 0.061
- Held-out test accuracy: 0.750

Best selected model:
- linear_svm
- CV accuracy: 0.650

Interpretation:
- Linear SVM outperformed Naive Bayes on both cross-validation and the held-out
  test set.
- The dataset is small, so class-level metrics are uneven.
- Some classes such as greeting and thanks performed poorly in Naive Bayes but
  better in Linear SVM.
- The hybrid architecture is important because the rule-based knowledge base
  compensates for the limited dataset.

11. Knowledge Base
------------------

The chatbot relies on data/knowledge_base.json for deterministic support.

Each category in the knowledge base can include:
- label
- keywords
- causes
- clarifying_questions
- solution_steps
- severity
- confidence_base

This file is what powers:
- keyword matching
- clarifying questions
- final troubleshooting steps
- severity labels

- Many hardware CSVs are spec tables, not symptom narratives.
- Those were skipped because they do not map naturally to user complaint text.
- Many image folders were cataloged as candidate labels, but the labels are
  structural guesses and need review.


13. Database Layer
------------------

database.py supports two backends:
- SQLite by default
- MySQL when DB_ENGINE=mysql is set

Tables:
- conversations
- feedback
- dialogue_sessions

Conversation logging records:
- session_id
- user_message
- fault_category
- confidence
- method
- source
- timestamp

Dialogue state logging makes the chat multi-turn and persistent within a session.


14. Web Application Routes
--------------------------

Defined in app.py:
- GET /
  Main chat UI
- POST /api/chat
  Accepts a message, optional image_finding, and conversation history
- POST /api/feedback
  Records helpful/not helpful feedback
- POST /api/reset
  Clears the dialogue state for the session
- GET /api/history
  Returns recent conversation log
- GET /api/stats
  Returns fault frequency counts
